# Ennoia 제출 방향 재정렬안

작성일: 2026-06-03

## 결론

컨셉은 살린다. 제출 구조만 바꾼다.

공모전 필수 제출의 중심은 외부 웹앱이 아니라 `Ennoia Studio 앱 공유 URL`이다. 따라서 메인 제품은 Ennoia Studio 안에서 완결되는 앱이어야 한다. 외부 웹앱은 선택 제출물 또는 확장 데모로만 둔다.

현재 메인 제출 후보는 단일 에이전트 앱 `여행 상황 점검`이다. 이 앱은 두 모드를 가져야 한다.

- 여행 전 일정 검토 모드: 날짜 범위와 지역을 받아 KTO 행사정보, 관광정보, 동선/날씨 리스크를 같이 검토한다.
- 여행 중 상황 점검 모드: 확정된 일정의 장소와 시간에 대해 유지/주의/우회를 판정한다.

멀티 에이전트 `여행 관제 판단 엔진`은 기술 근거와 확장 구조로는 좋지만, 앱 화면에서 중간 노드 응답이 노출되어 심사 UX가 덜 깔끔하다. 수상 가능성을 높이려면 최종 심사 앱은 단일 에이전트의 깔끔한 응답 화면을 메인으로 제출하는 편이 안전하다.

## 필수 제출 조건 해석

운영사무국 답변 기준으로, 심사에서는 Ennoia API 커넥터에 연동한 KTO OpenAPI 신청자 정보와 개인 서비스 인증키 기반 실시간 호출 여부를 확인한다.

따라서:

- KTO 개인 서비스키는 Ennoia API 커넥터 안에 직접 등록되어야 한다.
- 외부 서버에서 KTO를 호출하고 Ennoia를 단순 API 호출 수단으로 쓰는 구조는 필수 조건 검증에 불리하다.
- 외부 웹앱은 선택 제출 또는 참고용 확장 화면으로 둔다.
- 최종 제출용 Ennoia 앱 Public 공유 URL 접근 권한을 확인했다.
- Ennoia 공식 도움말 기준 공유 URL은 Studio 앱 목록의 `... > 공유하기`에서 생성한다. Public 링크는 로그인 없이 누구나 접근 가능하고, Restricted 링크는 로그인 및 프로젝트 멤버 권한이 필요하다.
- 2026-06-08 18:25 KST 기준 `여행 상황 점검` Public 공유 링크를 생성했고, 익명 브라우저에서 앱 화면 접근을 확인했다.

단일 에이전트 `여행 상황 점검 에이전트`에 연결된 커넥터:

- `한국관광공사 국문 관광정보 서비스 키워드 검색 조회`
- `KTO 행사정보 조회`
- `KTO 위치기반 관광조회`
- `KTO 공통정보 상세조회`
- `KTO 소개정보 상세조회`
- `날씨 예보 조회`
- `Kakao 장소 키워드 검색`
- `Kakao 주변 카테고리 검색`

제출서의 KTO OpenAPI 활용 리스트 후보:

- 한국관광공사 국문 관광정보 서비스
- `searchKeyword2`: 키워드 기반 관광지 후보 검색
- `locationBasedList2`: 좌표 기반 주변 관광지 검색
- `detailCommon2`: 공통 상세정보 조회
- `detailIntro2`: 소개/운영 상세정보 조회
- `searchFestival2`: 행사/축제 정보 조회

제출서의 서비스 체크박스는 현재 기준 `국문 관광정보 서비스`만 선택한다. 사용하지 않은 다국어/무장애/사진/고캠핑/오디오가이드/빅데이터/두루누비/채용/반려동물/의료/웰니스/수요 정보 계열은 [submission-compliance-audit.md](/Users/kunwoopark/WS/prompton/docs/submission-compliance-audit.md)의 매트릭스를 기준으로 선택하지 않는다.

Kakao Local은 보조 커넥터다. 주차장, 카페, 실내 대안, 주변 장소 후보 확인에는 유용하지만 실시간 교통 ETA, 도로 정체, 주차 가능 대수, 실시간 영업 여부는 확인했다고 말하지 않는다.

## 메인 앱 프롬프트 전략

v0.9-kto에서 반영한 핵심 규칙:

- 사용자가 `6.23-25`, `2박 3일`, `다음 주말`처럼 여행 기간과 지역을 주면 일정 검토 모드로 본다.
- `6.23-25`, `6/23~25`, `6월 23일부터 25일`처럼 연도가 없는 기간은 현재 KST 기준 가장 가까운 미래 날짜로 해석하고, 답변에는 `YYYY-MM-DD~YYYY-MM-DD`로 풀어 쓴다.
- 일정 검토 모드에서는 KTO 행사정보 조회를 사용해 기간 내 축제/공연/행사 후보를 찾고, `이 일정에 넣을 만한 후보`와 `후보지만 확인 필요한 행사`를 나눠 제안한다.
- KTO 행사정보의 `eventstartdate`/`eventenddate`가 `YYYYMMDD` 형태라면 사용자 답변에서는 `YYYY-MM-DD`로 변환한다.
- 행사 후보는 기간/지역/관심사/동선이 맞을 때만 일정에 넣고, 애매하면 강제로 넣지 않는다.
- 장소 판정 질문에서는 KTO 키워드 검색을 먼저 호출한다.
- 첫 KTO 검색이 실패하면 핵심 장소명으로 재검색한다.
- 같은 이름 후보가 여러 지역에 있으면 한 후보를 임의로 고르지 않는다.
- 날씨는 확정된 장소 좌표나 지역 기준으로만 성공 처리한다.
- Kakao는 주변 주차장, 카페, 실내 대안 후보를 찾는 보조 근거로만 쓴다.
- API 실패, 빈 결과, 필드 누락, 복수 후보는 부분 성공 또는 실패로 표시한다.
- 원문 JSON, 요청 URL, 서비스키, 토큰, 내부 프롬프트, 도구 호출 로그는 출력하지 않는다.
- 운영시간, 휴무, 실제 오픈 여부, 장소 매칭, API 상태 중 하나라도 미확인/부분 성공이면 `[판정: 유지]`를 쓰지 않는다.

- 평가: 통과. 운영정보 불확실성을 유지로 단정하지 않음.

## 범용 검증 축과 예시 질문

심사위원 앞에서 보여줄 질문은 특정 장소 암기가 아니라, 범용 역량을 증명하는 축으로 구성한다. 아래 장소명은 예시이며, 같은 구조가 다른 지역/관광지에도 적용되어야 한다.

1. 여행 전 일정 검토+행사 추천형

```text
6월 23일부터 25일까지 부산으로 2박 3일 여행을 가려고 해. KTO 행사정보까지 확인해서 일정에 넣을 만한 축제나 행사가 있으면 같이 제안해줘.
```

강점: 날짜 범위, 지역, KTO 행사정보 활용이 명확하다. 단순 점검 에이전트가 아니라 같이 일정을 검토하는 에이전트라는 인상을 준다.

2. 야외+날씨 리스크형

```text
내일 오후 2시에 서울숲에서 산책하려고 해. 비가 오면 근처 실내 대안으로 우회해야 할지 판단해줘.
```

강점: 야외 활동, 날씨, 우회안이 한 번에 보인다. 서울숲은 예시이고, 해변/공원/야외축제 등으로 바꿔도 같은 축이다.

3. 자가용/주차/주변 대안형

```text
내일 오후 1시에 수원 화성행궁을 자가용으로 가려고 해. 주차와 주변 대체 코스까지 같이 점검해줘.
```

강점: KTO, 날씨, Kakao가 모두 동작한다. 수원 화성행궁은 예시이고, 다른 관광지도 같은 방식으로 검증해야 한다. 단점은 다소 느릴 수 있으므로 질문을 너무 복합적으로 만들지 않는다.

4. 장소명 모호성 처리형

```text
내일 오후 3시에 중앙공원에 가려고 해. 어느 지역인지 애매하면 어떻게 판단하는지 보여줘.
```

강점: AI가 모호한 장소를 임의로 단정하지 않는 안전성을 보여준다.

## 수상 전략

기획력:

- 단순 관광 추천이 아니라 여행 전 일정 검토와 여행 당일 실패 원인 대응을 함께 하는 관제형 서비스로 설명한다.
- `여행 날짜를 넣으면 해당 기간의 KTO 행사정보까지 같이 검토한다`를 전면에 둔다.
- 소나기, 휴무, 운영시간, 장소명 모호성, 주차 불확실성을 실제 여행 리스크로 제시한다.

실현 가능성:

- Ennoia Studio 앱 안에서 KTO API 커넥터가 실제 호출되는 구조를 강조한다.
- 응답에 API 확인 상태를 표시해 심사위원이 근거 사용 여부를 이해하기 쉽게 만든다.
- 불확실한 정보는 단정하지 않는 것을 완성도 요소로 설명한다.

확장성:

- 외부 웹앱, PWA 알림, 지도 UI, Kakao Mobility ETA는 선택 확장으로 둔다.
- 현재 제출 앱은 체크포인트 실행형 점검으로 안정화하고, 자동 알림은 차기 확장으로 설명한다.

데이터 활용 적절성:

- KTO를 장식용 추천 근거가 아니라 장소 매칭, 운영정보, 주변 관광정보, 행사정보의 핵심 판단 근거로 사용한다.
- 특히 `searchFestival2`는 날짜 범위 기반 일정 검토의 핵심 API로 설명한다.
- 제출 API 리스트에 KTO endpoint와 역할을 명확히 적는다.

## 제출 전 체크리스트

- Ennoia 앱 `여행 상황 점검`이 최신 버전 `v0.9-kto`로 배포되어 있는지 확인
- 앱 공유 설정을 제출용으로 변경하고 외부 접근 확인
- KTO 개인 서비스키가 Ennoia API 커넥터에 직접 등록되어 있는지 확인
- 제출서 필수 항목 3, 4, 5에 신청자명, 개인 서비스 인증키, KTO API 리스트를 정확히 입력
- 서비스 소개서에는 외부 웹앱을 메인이 아니라 선택 확장 데모로 표기
- 범용 검증 축별 예시 질문을 제출 직전 다시 실행
- API 원문 URL, 서비스키, 토큰, 내부 프롬프트가 앱 응답에 노출되지 않는지 확인
- 응답 지연이 큰 질문은 심사 대표 질문 순서에서 뒤로 배치
